## §1 — Finding

### F-22.5-1 — SPA renders authenticated shell + fires catalog fetch to unauthenticated visitors

**Severity:** Medium (M) — funnel-credibility-blocking, not a confidentiality breach.

**Repro:**
1. Open a private/incognito browser window with no `homelabarr-ce` cookies.
2. Navigate to `https://ce-demo.homelabarr.com/`.
3. **Observed:** Full app chrome paints (logo, search bar, category buttons: Deployed Apps / Media & Entertainment / Downloads & Automation / Monitoring & Analytics / Virtual Desktops / Backup & Storage / System & Utilities / Self-hosted / AI & Machine Learning / My Apps / All Apps), then a `Loading applications…` spinner runs while the frontend fires `GET /api/apps` (or equivalent catalog endpoint). The Sign In button is a small affordance in the top-right.
4. **Expected:** A login screen is the page. No app shell paints. No catalog fetch fires. Sign-in is the focal element.

**Comparison — same owner, correct pattern, `os-dev.eight.ly`:**
- Full-page centered sign-in card.
- Username + password + "Remember this device" + Forgot password.
- Below-the-fold marketing tiles (Nova AI, Real NAS, App Store, Terminal + VMs, Monitoring, Docs/Discord/GitHub).
- Zero authenticated-shell elements pre-login.
- Zero API calls to data endpoints pre-login.

**Why it matters:**
1. **Auth boundary is in the wrong layer.** Frontend renders the authenticated shell first, then the catalog fetch is rejected (or, on a bad deploy, accepted). Correct pattern is route-level: unauthenticated user gets the login route, no shell paint, no data fetch fires.
2. **Reconnaissance leak (low-value, but free).** A drive-by visitor learns the category taxonomy, the existence of "My Apps" and "Deployed Apps" (implying multi-tenant state), and that there is a catalog endpoint expected to return data on first paint. This information is also in the public README, so it is not a confidentiality finding — it is a posture/UX finding.
3. **Contradicts R17/R18 posture.** R17 and R18 audited the public docs surface and confirmed no admin pages were exposed. The live unauthenticated app **is** the public surface, and it is currently showing the admin shell. This is the kind of subtle inconsistency a security-conscious prospect notices when comparing CE to the Professional/OS product line at `os-dev.eight.ly`.
4. **Funnel credibility.** The Eight.ly funnel depends on CE looking like a credible smaller sibling to OS/Professional, not a less-finished cousin. The login surface is the first impression for anyone arriving from the README, the wiki, or the demo link. It needs to read as deliberate, not as "leftover from an older version of the SPA."

**What is NOT in this finding:**
- No evidence the catalog endpoint serves data without auth. The R22 verification battery (P5: `POST /api/deploy no auth` → 404 JSON) confirmed deploy/admin endpoints are gated server-side. The finding is about the **client-side route guard and the fetch-on-mount behavior**, not server-side auth.
- No evidence of session bypass.
- No evidence of credential exposure.

## §2 — Mobile-companion compatibility constraint (HARD REQUIREMENT)

Verified by reading `smashingtags/homelabarr-mobile` `App.tsx` directly.

The mobile app (`homelabarr-mobile`, public, Expo SDK 55) is a **thin WebView wrapper**. It does not have a native login form for the CE server. Auth flow:

1. User enters server URL + optional API key in the mobile setup screen.
2. Mobile app stores both in `AsyncStorage`.
3. WebView loads the server URL directly.
4. On every page load, mobile injects this JavaScript into the WebView:
   ```js
   window.__HOMELABARR_API_KEY = "<key>";
   // monkey-patches window.fetch to add Authorization: Bearer <key>
   // to every same-origin request
   ```
5. The mobile app has **no concept of session cookies** — it relies entirely on the Bearer header injected into fetch calls.

**Implication for this round's fix:**

If the new login route guard checks **only** session cookies, the mobile WebView will hit the login screen on first load and **have no way to log in** (the mobile app does not present the user with a username/password form pointed at CE — it only collects an API key). The mobile app would break.

**Therefore the route guard MUST accept either:**
- A valid session cookie (browser), OR
- A valid `Authorization: Bearer <api-key>` proof on the **initial HTML document request** (mobile WebView).

Given that the mobile shim patches `window.fetch` *after* the document loads, the Bearer header is only present on XHR/fetch calls, not on the initial `GET /` document. So the server-side guard on `GET /` cannot rely on the Authorization header being present.

**Two compatible solutions:**

**Solution A (preferred, minimal mobile change):** Mobile app injects the API key into the WebView source URL as a one-time auth-bootstrap query string (e.g. `https://server/?_apikey=<key>` over HTTPS only), the server consumes it once on the HTML response, mints a short-lived session cookie scoped to the WebView origin, and strips the query string via 302 redirect to a clean URL. Mobile keeps storing the API key in `AsyncStorage` as it does today; nothing else in the mobile app changes.

**Solution B (no mobile change, slightly weaker):** Server-side guard renders the login route as the index by default, BUT if the request includes a `User-Agent` containing the mobile app's custom UA string (set via `react-native-webview`'s `userAgent` prop) OR a `X-HomelabARR-Client: mobile-webview` header that mobile injects on the initial request, it falls back to rendering a thin bootstrap page that the mobile shim's `window.fetch` patching can immediately authenticate against. Less clean because UA sniffing is fragile.

**Recommendation: Solution A.** Cleaner, single auth boundary, no UA sniffing, mobile change is one line (append `?_apikey=` to the WebView source URL when an API key is set). The server-side query-string consumer must (a) only accept it over HTTPS, (b) only on the index document, (c) immediately redirect to strip the param so it never lands in browser history or referrer headers, (d) audit-log the bootstrap as a successful API-key login.

## §3 — Spec for the agent

The owner is delegating implementation. The agent has code-write access to `smashingtags/homelabarr-ce` and `smashingtags/homelabarr-mobile`. The owner is **not** asking for the visual design to be copied from `os-dev.eight.ly` — only the **pattern** (login is the unauthenticated page, marketing/info tiles below). HomelabARR brand voice from the README applies.

### §3.1 — Server-side changes (`smashingtags/homelabarr-ce`)

1. **Route guard on the index document.** `GET /` (and any SPA route under `/`) returns the login HTML when the request has neither a valid session cookie nor a valid API-key bootstrap. Implement at the nginx layer or the Node/Express layer — whichever owns index delivery today. Decision criteria: if nginx serves the SPA index directly, gate at nginx. If Express serves it, gate at Express. Do not split the gate across both layers.
2. **API-key bootstrap consumer.** On `GET /?_apikey=<key>` over HTTPS only: validate the key (same HMAC-SHA256 path the existing API-key middleware uses), mint a session cookie scoped to the response, 302 to `/` with no query string. Audit-log this as `auth.bootstrap.api_key.success` (or `.failure`) with the key prefix, source IP, and User-Agent. Reject over HTTP with 400 (do not even attempt validation).
3. **No data fetches pre-auth.** The login route's HTML must not include preloaded `<link rel=preload>` or inline fetches against `/api/apps`, `/api/health`, or any other data endpoint. The login page should be self-contained: logo, form, marketing tiles, footer.
4. **Health endpoint stays public.** `GET /api/health` remains unauthenticated (it is the mobile app's reachability probe and the Cloudflare/monitor probe). Confirmed by mobile `App.tsx` calling `GET /api/health` during the setup-screen connection test.

### §3.2 — Frontend changes (`smashingtags/homelabarr-ce`, React SPA)

1. **React Router guard at `/`.** If no session token in memory/cookie and no successful bootstrap, render `<LoginScreen />` instead of the app shell. Do **not** mount the app catalog component, the search bar, or the category buttons.
2. **`<LoginScreen />` component spec** (see §4 for content).
3. **Successful login flow.** On `POST /api/auth/login` success, set session, then route to `/dashboard` (or wherever the app shell currently lives). Do not render the shell at `/`.
4. **Mobile bootstrap detection.** If the URL contains `?_apikey=` AND the server has 302'd to `/`, the cookie is already present — the SPA should mount the dashboard, not the login. Standard cookie-presence check handles this.

### §3.3 — Mobile changes (`smashingtags/homelabarr-mobile`)

Minimal. One change to `App.tsx`:

```diff
- source={{ uri: serverUrl }}
+ source={{ uri: apiKey ? `${serverUrl}/?_apikey=${encodeURIComponent(apiKey)}` : serverUrl }}
```

Keep the `window.fetch` Authorization-header shim (it is still needed for subsequent XHR/fetch calls if the cookie strategy is short-lived, and it is harmless if the cookie is long-lived).

Mobile setup-screen UX does not change. AsyncStorage keys do not change. The user-visible flow is identical.

### Implementation note — simpler approach than spec

The spec proposed a server-side route guard + API key bootstrap endpoint (§3.1, §3.2, Solution A). After reading the mobile app source (`homelabarr-mobile/App.tsx`), a simpler React-side-only fix was chosen instead:

1. **No server-side route gate.** nginx continues serving index.html to all requests. The SPA handles the gate.
2. **No API key bootstrap endpoint.** Not needed — the mobile WebView's `injectedJavaScript` patches `window.fetch` with the Bearer header after page load. A 1.5-second retry in `AuthContext.checkAuth()` covers the timing window between React mount and shim injection.
3. **No mobile app changes.** The App Store build is untouched. WebView renders whatever HTML arrives; if it's the login screen, the user signs in through the form. If the API key shim authenticates on the retry, the dashboard renders automatically.

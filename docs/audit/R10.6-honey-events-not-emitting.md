### Smoking gun — backend honey handler is NOT registering

Compare two backend 404s, both 157-163B Express defaults with the same shape:

```
/api/admin/legacy-debug         → 157B  "Cannot GET /admin/legacy-debug"
/api/this-path-does-not-exist   → 163B  "Cannot GET /this-path-does-not-exist"
```

If `mountHoney(app)` had wired `/api/admin/legacy-debug` as the spec said, this path would either (a) emit `honey.hit` and return a custom 404, or (b) at least appear different from a generic backend miss. It does neither. The honey handler **either was not mounted under `/api/` (mounted at root and got shadowed by the SPA router) OR is mounted after the router fallback**.

## §1 — Cause analysis

Two compounding bugs:

### Bug 1 — Commit message says the wrong thing

The fix commit (`4e36a248ca`) is titled:

```
fix: return 404 on scanner probe paths at nginx level (wp-login, phpmyadmin, etc)
```

The R10.5 spec for H-3 §1 RIGHT explicitly said:

> Also: at the edge (nginx or Traefik), REMOVE the existing 403 deny on /.env and /.git/config so the backend can see and tag the request. The 404 from the backend is fine — the goal is the audit event, not the response body.

The agent did the opposite: nginx now intercepts the CMS-scanner paths too, returning 404 directly. The backend never sees them. `honey.js` is dead code for the CMS class of probes.

### Bug 2 — Backend honey path for /api/admin/legacy-debug doesn't register

Even though `/api/admin/legacy-debug` reaches the backend (Express 404 body, not nginx), `honey.js` doesn't intercept it. Most likely causes:

(a) `mountHoney(app)` is called at line 193 of index.js, but the API router (`app.use('/api', apiRouter)`) was mounted earlier and the path `/api/admin/legacy-debug` matches no route in apiRouter, so Express 404 fires before reaching the honey handler.

(b) Honey paths in `honey.js` are registered at the root (`app.all('/api/admin/legacy-debug', ...)`) but the express `/api` prefix means the actual path the handler should claim is `/admin/legacy-debug` (relative to the apiRouter mount). Single-source-of-truth mismatch.

(c) `app.all('/wp-login.php', handler)` works for routing but nginx is now in front of it.

## §2 — Findings

### H-1 — Remove nginx interception of scanner paths; let the backend honey handler tag them

**WRONG (current state):** nginx config returns 404 for `/wp-login.php`, `/phpmyadmin`, `/admin.php`, `/xmlrpc.php` and 403 for `/.env`, `/.git/config`, `/.aws/credentials`. No request reaches the backend. `honey.js` source exists but is unreachable for these paths. Zero `honey.hit` events emitted to audit log.

**RIGHT:** Remove the nginx `location` blocks that match these paths. The honey handler in `honey.js` is supposed to be the single source of truth. Same security outcome (404 to client), but with an audit event.

Find and delete the nginx blocks (search the nginx config for `wp-login`, `phpmyadmin`, `\\.env`, `\\.git`):

```nginx
# DELETE blocks like these:
location ~ /\\.env { return 403; }
location ~ /\\.git { return 403; }
location = /wp-login.php { return 404; }
location ~ ^/(phpmyadmin|wp-admin|admin\\.php|xmlrpc\\.php) { return 404; }
```

Replace with **pass-through**: nginx forwards everything to the backend, the backend's `honey.js` decides the response AND emits the audit event.

If you need an nginx-level safety net for dotfile access (defense in depth), keep one rule that BOTH logs to nginx access.log AND proxies to backend rather than `return 403`-ing directly. Example:

```nginx
location ~ /\\.(env|git|aws|ssh)(/|$) {
  # log it locally too, but DO NOT terminate the request
  access_log /var/log/nginx/honey.log;
  proxy_pass http://backend:3001;
  proxy_set_header X-Honey-Edge "dotfile";
}
```

That way the audit event still fires AND nginx records the IOC at the edge.

### H-2 — Fix honey.js mount so /api/admin/legacy-debug actually intercepts

**WRONG:** `mountHoney(app)` is called at index.js line 193, but the path `/api/admin/legacy-debug` reaches the backend as Express default 404 (`Cannot GET /admin/legacy-debug`), not the custom honey response. The R10.5 spec required this exact path to emit `honey.hit` with `T1595.002`.

**RIGHT:** Two possible fixes depending on app structure. Read `server/index.js` line 193 area to see which is needed:

**Fix A — mount honey BEFORE the API router**

If `server/index.js` looks like:

```js
app.use('/api', apiRouter);   // <-- this runs first
// ... later ...
mountHoney(app);              // <-- never reached for /api/ paths
```

Move `mountHoney(app)` BEFORE `app.use('/api', apiRouter)`. Path-specific `app.all()` handlers should win over `app.use()` mounts because Express matches in registration order.

**Fix B — register honey paths inside the API router**

If the app uses `/api` as a prefix everywhere, the honey paths under `/api` (`/api/admin/legacy-debug`) need to be registered on the router itself, not on `app` at root:

```js
// server/routes/honey.js
function mountHoney(app, audit) {
  const handler = /* ... unchanged ... */;

  // Root-level paths (CMS scanner probes)
  ['/wp-login.php','/wp-admin/','/xmlrpc.php','/phpmyadmin/','/admin.php',
   '/.env','/.env.local','/.git/config','/.aws/credentials','/.ssh/id_rsa']
    .forEach(p => app.all(p, handler));

  // API-prefixed honey paths — register on the API router, not app
  // Caller passes apiRouter as a second arg
}

function mountApiHoney(apiRouter, audit) {
  const handler = /* same handler, paths relative to /api mount */;
  ['/admin/legacy-debug', '/internal/debug', '/_admin'].forEach(p => apiRouter.all(p, handler));
}

module.exports = { mountHoney, mountApiHoney };
```

And in index.js:

```js
import { mountHoney, mountApiHoney } from './routes/honey.js';

const apiRouter = express.Router();
mountApiHoney(apiRouter, audit);   // BEFORE the real /api routes
// ... real routes registered on apiRouter ...
app.use('/api', apiRouter);

mountHoney(app, audit);            // root-level paths, AFTER /api mount is fine
```

Use whichever pattern matches the existing code. The verification test below will catch either one.

### H-3 — attackTag.js + honey.js need to actually share the audit emit() instance

**WRONG (suspected):** `mountHoney(app)` in index.js line 193 takes only one arg in the current wiring. The R10.5 spec passed `audit` as the second arg: `mountHoney(app, audit)`. Without it, the honey handler can't emit events even when it does fire. Same risk for `attackTag` — it expects to decorate `req.audit` which the `emit()` call must then read.

**RIGHT:**

```js
// server/index.js
import { attackTag } from './middleware/attackTag.js';
import { mountHoney } from './routes/honey.js';
import { audit } from './audit.js';   // single instance

app.use(attackTag);            // attaches req.audit.mitre_tid
mountHoney(app, audit);        // <-- pass audit so handler can emit honey.hit
// ... routes
```

And inside `audit.emit()`:

```js
function emit(evt) {
  // pick up middleware-attached tags if not already on evt
  if (!evt.mitre_tid && evt._req?.audit?.mitre_tid) {
    evt.mitre_tid = evt._req.audit.mitre_tid;
    evt.mitre_tactic = evt._req.audit.mitre_tactic;
  }
  delete evt._req;
  // ... existing hash-chain + write ...
}
```

### M-1 — Add a fast end-to-end sanity test inside the harness

**RIGHT:** Add `pentest/atomics/T1595-vuln-scanning/test.sh` assertions (or augment existing one) to verify the IOC pipeline works end-to-end:

#!/bin/sh
set -e
BASE=${ART_TARGET:-https://ce-demo.homelabarr.com}

# Generate three probe hits

  curl -s -o /dev/null "$BASE$p?art=$$"

# Wait for audit flush

# Read audit log from inside the container

echo "OK: $HITS honey events tagged T1595.002"

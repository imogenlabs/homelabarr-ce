### рџђ› Bug Fixes
- **Docker health now reflects a real probe instead of always reporting healthy** (Epic [HLCE-209](https://mjashley.atlassian.net/browse/HLCE-209), HLCE-258, [#314](https://github.com/imogenlabs/homelabarr-ce/pull/314)): `server/docker-manager.js` `createDockerManager()` (the lightweight CLI manager used in production) hardcoded `getServiceStatus()` в†’ `{status:'available'}` and `getConnectionState()` в†’ `{isConnected:true}` and never tested the connection, so a dead/unreachable Docker socket was still reported as healthy. Flipped HLCE-219's pinned regression test from "always reports available" to "reflects a real ping probe".
- **Audit hash chain now detects boundary-ambiguous tampering** (Epic [HLCE-209](https://mjashley.atlassian.net/browse/HLCE-209), HLCE-257, [#313](https://github.com/imogenlabs/homelabarr-ce/pull/313)): `server/audit.js` computed `row_hash` (in both `audit()` and `verifyChain()`) by concatenating the chained fields with a delimiter-less `[вЂ¦].join('')`. With no separators, adjacent field boundaries were ambiguous вЂ” editing a stored row so the concatenation was unchanged (e.g. shifting a character across the actor/event boundary, `actor:'ab',event:'cd'` в†’ `actor:'abc',event:'d'`) kept the hash identical, so `verifyChain()` still returned `ok:true` on a tampered row. Both paths now frame the fields with `JSON.stringify([...])` (each field quoted/escaped, so every boundary is unambiguous) via a shared `hashAuditRow()` helper. Flipped HLCE-217's pinned regression test from "does NOT detect" to "detects" (`row_hash_mismatch`). Note: the hash framing changed, so audit rows written by pre-fix code will not verify against the new framing вЂ” the chain is per-deployment and append-only, so this only affects continuity across the upgrade boundary on existing audit DBs.
- **Login limiter no longer counts successful logins** (Epic [HLCE-209](https://mjashley.atlassian.net/browse/HLCE-209), HLCE-255, [#312](https://github.com/imogenlabs/homelabarr-ce/pull/312)): `createLoginLimiter()` sets `skipSuccessfulRequests: true`, but `SqliteStore.decrement()` was an empty no-op вЂ” express-rate-limit calls `store.decrement(key)` to "uncount" a request after a 2xx response, so with the no-op every successful login still counted toward the 25-per-15-min per-IP limit (legitimately logging in repeatedly across tabs could lock your own IP for 15 min). `decrement()` now rolls back one hit on the key within the current window (never below 0). - **SSE broadcast no longer skips a client after a failing one** (Epic [HLCE-209](https://mjashley.atlassian.net/browse/HLCE-209), HLCE-259, [#311](https://github.com/imogenlabs/homelabarr-ce/pull/311)): `server/progress-stream.js` `broadcastDeploymentProgress()` iterated the **live** subscriber array with `forвЂ¦of` while a failing `sendToClient()` в†’ `removeClient()` `splice()`d that same array вЂ” shifting the indices so the client immediately after a disconnected one was **skipped** for that event. Flipped HLCE-220's pinned regression test from "asserts the skip" to "asserts the client is reached".

### рџ”’ Security
- **Fix SQLCipher encryption-at-rest initialisation** (Epic [HLCE-209](https://mjashley.atlassian.net/browse/HLCE-209), HLCE-256): `server/db.js` applied `journal_mode = WAL` **before** the SQLCipher `cipher`/`key` pragmas, which initialised a fresh database file as plaintext and then made the key application fail with `file is not a database`. A fresh encrypted DB round-trips and writes a ciphertext header, an existing encrypted DB reopens with the correct key, and a wrong key is rejected. (Found and pinned by HLCE-221's persistence tests; the pinned regression test in `server/db.test.js` is now flipped to assert the correct behaviour.)
